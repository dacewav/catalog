## Análisis profundo del repo

### Arquitectura general
- **Store** (`src/main.js` → `dist/store-app.js`): Frontend público
- **Admin** (`src/admin-main.js` → `dist/admin-app.js`): Panel de administración
- **Shared**: `src/card-style-engine.js` usado por ambos
- **Deploy**: Cloudflare Pages (lee del repo, sirve desde root)
- **Build**: `node build.js` → esbuild → dist/ + hash cache busting

### Módulos admin (43 archivos en src/admin/)
- **Core hub**: `core.js` — wiring de dependencias, re-exports, window assignments
- **State**: `state.js` — exports de variables mutables (db, allBeats, T, etc.)
- **Firebase**: `firebase-init.js` — auth, data loading, initAdmin()
- **Preview sync**: `preview-sync.js` — postMessage al iframe del store
- **Beats**: `beats.js` — CRUD de beats, saveBeat()
- **Card style**: `beat-card-style.js` — build/apply cardStyle desde inputs
- **Card global**: `card-global.js` — estilo global de tarjetas
- **Card effects**: `card-effect-presets.js` — presets de efectos (FX gallery)
- **Theme IO**: `theme-io.js` — collectTheme/loadThemeUI
- **Autosave**: `autosave.js` — auto-save con debounce

### Módulos store (20 archivos en src/)
- **Main**: `main.js` — entry point, boot sequence
- **Cards**: `cards.js` — renderAll, beat card HTML generation
- **Live edit**: `live-edit.js` — recibe postMessage del admin
- **State**: `state.js` — state global de la tienda
- **Theme**: `theme.js` — applyTheme
- **Card style engine**: `card-style-engine.js` — SHARED con admin

### Pares críticos (admin ↔ store)
1. **preview-sync.js ↔ live-edit.js**: postMessage communication
   - Admin envía: `admin-batch-update`, `beat-update`, `beat-revert`, `global-card-style-update`, `highlight-element`
   - Store envía: `index-ready`, `beat-update-ack`, `element-clicked`
   - **BUG**: `live-edit.js` handler de `global-card-style-update` SOBRESCRIBE cardStyles individuales de TODOS los beats

2. **beat-card-style.js ↔ card-style-engine.js**: build/apply cardStyle
   - Admin usa `buildCardStyle('f-', true)` para construir desde inputs
   - Store usa `mergeCardStyles(globalCs, beatCs, beatHasCustom)` para renderizar

3. **beats.js ↔ Firebase ↔ main.js**: CRUD de beats
   - Admin guarda en `Firebase/beats/{id}`
   - Store lee de `Firebase/beats` y renderiza

4. **autosave.js ↔ Firebase ↔ live-edit.js**: auto-save de theme/settings
   - Admin guarda theme/settings en Firebase
   - Store recibe actualizaciones por postMessage (más rápido que Firebase listener)

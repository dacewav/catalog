## Bug #1: Panel overlap — PERSISTE (no resuelto)

**Status:** NO resuelto después de 3 intentos de fix.

**Síntoma:** Los cards de Extras ("Glow animado", "Animación de tarjeta", etc.) aparecen visibles en el tab Plataformas (y probablemente otros tabs). Específicamente: tab "Plataformas" activo pero se ven cards que pertenecen a `etp-extras`.

**Intentos de fix:**
1. CSS `!important` en `.etp { display: none !important; }` — no bastó
2. Backup CSS: `.etp:not(.on) > * { visibility: hidden !important; }` — no bastó  
3. JS hardening en `showEt()`: force reflow + defensive hide `anim-subsettings` — no bastó

**Lo que NO es el problema:**
- La lógica JS de `showEt()` es correcta (remove all `.on`, add to target)
- No hay `display: block !important` que compita (solo `.etp.on`)
- `anim-subsettings` está correctamente dentro de `etp-extras`
- No hay duplicados de IDs

**Hipótesis sin confirmar:**
- Posible: el CSS dist no se está recargando (caché de Cloudflare/browser). Necesita busting o hard refresh.
- Posible: algún CSS inline en `admin.html` que sobreescribe `display: none` en los `.etp`.
- Posible: el `section-in` animation (opacity 0→1 + translateY) causa que el contenido de múltiples `.etp` se renderice durante la animación y quede visible.
- **Más probable:** revisar si `sec-add` tiene `overflow: visible` y si algún `.etp` tiene `position: absolute` que lo saque del flow normal, haciendo que `display: none` del padre no lo oculte.

**Para debuggear en browser:**
1. Abrir admin → DevTools → inspeccionar un `.etp` que debería estar oculto
2. Verificar en Computed Styles si `display: none` está realmente aplicado
3. Si no, buscar qué regla CSS lo está sobreescribiendo
4. Verificar si `_headers` tiene algún header que cache el CSS

## Bug #2: Galería de imágenes vacía — no carga datos existentes

**Status:** Implementado pero NO funciona con beats existentes.

**Síntoma:** Beat "001-fantasma" tiene imagen (se ve thumbnail en la lista), pero al abrir el editor la galería muestra "Sin imágenes".

**Causa probable:** El beat en Firebase no tiene el campo `images: []` (feature nueva). El fallback `b.imageUrl ? [b.imageUrl] : []` debería cargar la imagen existente como galería de 1, pero no lo está haciendo.

**Debuggear:**
1. Verificar si `b.images` existe como array vacío `[]` en Firebase (condición `b.images && b.images.length` fallaría)
2. Verificar si `b.imageUrl` es truthy en el momento del load
3. Agregar `console.log` en openEditor: `console.log('images:', b.images, 'imageUrl:', b.imageUrl, '_imgGallery:', _imgGallery)`
4. Posible fix: cambiar condición a `if (b.images && Array.isArray(b.images) && b.images.length > 0)`

**También:** `prevImg()` se llama DESPUÉS de cargar los datos del beat, pero `_imgGallery` se setea ANTES de `prevImg()`. Si `prevImg()` agrega `val('f-img')` a `_imgGallery` de nuevo, debería funcionar... a menos que `val('f-img')` retorne vacío en ese momento (timing issue con `setVal`).
